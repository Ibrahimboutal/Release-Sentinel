# Release Sentinel Artifacts & Output Files

This guide explains the output files and artifacts generated by Release Sentinel.

## Overview

After running Release Sentinel, you'll find these key files:

| File | Purpose |
|------|---------|
| `artifacts/release_verdict.json` | Complete release decision with all evidence |
| `artifacts/run_history.jsonl` | Compact audit trail of all runs (one JSON object per line) |
| `data/change_manifest.json` | Input: The proposed change description |
| `data/coverage_map.json` | Input: Test-to-capability mapping |

## Release Verdict JSON

The main output file `artifacts/release_verdict.json` contains:

### Structure

```json
{
  "change_id": "chg-xxx-123",
  "generated_at": "2026-05-28T16:01:17.471946Z",
  "decision": "approve|needs_review|block",
  "risk": {
    "score": 0-100,
    "level": "low|medium|high|critical",
    "drivers": ["reason 1", "reason 2", ...],
    "confidence": 0.0-1.0
  },
  "plan": {
    "selected_test_sets": [...]
  },
  "executions": [
    {
      "test_set_key": "SMOKE",
      "execution_id": "exe-12345",
      "status": "passed|failed|timed_out",
      "test_results": [...]
    }
  ],
  "triage": {
    "total_tests": 15,
    "passed": 13,
    "failed": 2,
    "failures": [
      {
        "test_key": "TC-001",
        "category": "product_bug|test_fragility|test_data_issue|environment_issue|needs_human_review",
        "confidence": 0.8,
        "reasoning": "..."
      }
    ]
  },
  "release_notes": ["Summary of decision and next steps"],
  "evidence_links": [
    "https://uipath-cloud/executions/12345"
  ]
}
```

### Interpreting Decision

- **`approve`** - Change is safe to release based on:
  - Low risk score + all tests pass, OR
  - Medium risk score + comprehensive test pass with high confidence

- **`needs_review`** - Ambiguous result requiring human judgment:
  - Failures not clearly product bugs
  - Risk score doesn't clearly justify approve/block
  - Timeouts or inconclusive evidence
  - **Action**: Routed to Action Center for human review

- **`block`** - Change should NOT be released:
  - High/critical risk score + any test failures
  - Failures clearly indicate product bugs
  - Multiple failures in critical capabilities
  - **Action**: Return to development, investigate failures

### Risk Score Interpretation

- **0-25**: Low risk
  - Safe smoke test only
  - Minor changes, docs, comments
  - Example: "Updated README"

- **26-50**: Medium risk
  - Targeted regression recommended
  - Logic changes, new features
  - Example: "Add new claim routing path"

- **51-75**: High risk
  - Full regression suite needed
  - Changes to critical code paths
  - Example: "Update eligibility engine"

- **76-100**: Critical risk
  - Manual review + full regression
  - Changes to regulated decisions
  - Example: "Change payment routing logic"

### Risk Drivers Example

```json
"drivers": [
  "base change risk: 15",
  "requirement mentions eligibility: +18",
  "requirement mentions routing: +16",
  "changed file matches routing: +14",
  "risk tag customer_impact: +18",
  "capability eligibility_routing: +16"
]
```

Each driver shows **what** triggered the risk score and **how much** it contributed.

## Failure Triage Categories

When tests fail, Release Sentinel classifies them:

### product_bug (Confidence: High)

**What**: The code change introduced a real bug

**Evidence**:
- Expected output differs from actual
- Assertion failure in test logic
- Not flaky in historical data
- Failure is reproducible

**Action**: Fix the bug before release

**Example**:
```json
{
  "test_key": "TC-ELIGIBILITY-001",
  "category": "product_bug",
  "confidence": 0.9,
  "reasoning": "Assertion 'claim routed to straight_through' failed but expected 'adjuster_review'. Not seen before. Matches change to eligibility logic."
}
```

### test_fragility (Confidence: Medium)

**What**: The test is flaky, not a real bug

**Evidence**:
- UI selector is brittle (coordinates, IDs)
- Timing assumptions (sleeps, timeouts)
- Historical flakiness >35% (configurable)
- Passes on retry

**Action**: Fix the test, don't block release

**Example**:
```json
{
  "test_key": "TC-UI-004",
  "category": "test_fragility",
  "confidence": 0.7,
  "reasoning": "Click handler on button element times out. Test has failed 42% in last 10 runs. Likely selector issue, not code bug."
}
```

### test_data_issue (Confidence: Medium)

**What**: Test preconditions are broken

**Evidence**:
- Setup step failed
- Test data is stale
- External service unavailable
- Database not initialized

**Action**: Fix test data, don't release

**Example**:
```json
{
  "test_key": "TC-DATA-002",
  "category": "test_data_issue",
  "confidence": 0.8,
  "reasoning": "Setup failed: 'Policy ABC-123 not found in database'. Likely test data cleanup incomplete."
}
```

### environment_issue (Confidence: Medium)

**What**: Infrastructure problem, not code

**Evidence**:
- Network error connecting to service
- Resource limits exceeded
- Dependency service down
- Permission denied

**Action**: Fix environment, retry

**Example**:
```json
{
  "test_key": "TC-SERVICE-003",
  "category": "environment_issue",
  "confidence": 0.75,
  "reasoning": "Connection refused to Test Manager at 192.168.1.100:8080. Network/service issue, not code."
}
```

### needs_human_review (Confidence: Low)

**What**: Unclear what the failure means

**Evidence**:
- Ambiguous error message
- Multiple potential causes
- Newly introduced test
- Complex flow failure

**Action**: Route to expert for analysis

**Example**:
```json
{
  "test_key": "TC-COMPLEX-005",
  "category": "needs_human_review",
  "confidence": 0.4,
  "reasoning": "End-to-end flow failed in 'process_claim' function with 'Unexpected state: PENDING'. Multiple possible causes. Requires expert review."
}
```

## Run History (JSONL)

The `artifacts/run_history.jsonl` file contains one JSON object per line, one entry per run:

```jsonl
{"run_id":"run-001","change_id":"chg-001","decision":"approve","risk_score":22,"tests_run":15,"passed":15,"failed":0,"generated_at":"2026-05-28T10:00:00Z"}
{"run_id":"run-002","change_id":"chg-002","decision":"block","risk_score":78,"tests_run":30,"passed":25,"failed":5,"generated_at":"2026-05-28T11:30:00Z"}
{"run_id":"run-003","change_id":"chg-003","decision":"needs_review","risk_score":52,"tests_run":25,"passed":24,"failed":1,"generated_at":"2026-05-28T13:00:00Z"}
```

Use for:
- Dashboard run history
- Audit trail for compliance
- Trend analysis (risk over time)
- CI/CD integration logging

## Change Manifest (Input)

Example `data/change_manifest.json`:

```json
{
  "title": "Update eligibility routing for injury claims",
  "requirement": "Route high-value injury claims directly to senior adjuster for expedited review per new SLA",
  "changed_files": [
    "src/claimspilot/eligibility.py",
    "src/claimspilot/routing.py",
    "tests/test_eligibility.py"
  ],
  "affected_capabilities": [
    "eligibility_routing",
    "audit_evidence"
  ],
  "risk_tags": [
    "customer_impact",
    "regulated_decision",
    "new_logic"
  ],
  "pull_request_url": "https://github.com/company/repo/pull/42"
}
```

Fields:
- **title**: One-line summary of change
- **requirement**: Business requirement or spec text
- **changed_files**: List of modified files
- **affected_capabilities**: Which business capabilities change
- **risk_tags**: Risk indicators (customer_impact, compliance, security, etc.)
- **pull_request_url**: Link to PR for full context

## Coverage Map (Input)

Example `data/coverage_map.json`:

```json
{
  "project_key": "REL_SENTINEL",
  "test_sets": [
    {
      "key": "SMOKE",
      "name": "ClaimsPilot smoke release gate",
      "execution_type": "smoke",
      "capabilities": ["claim_intake", "audit_evidence"],
      "test_cases": [
        {
          "key": "TC-SMOKE-001",
          "name": "Claim intake creates audit event",
          "capability": "claim_intake",
          "automation": "ClaimsPilot.Smoke.IntakeAudit",
          "priority": "P0"
        }
      ]
    }
  ],
  "capability_map": {
    "eligibility_routing": ["ELIGIBILITY_TARGETED", "CLAIMS_TARGETED_REGRESSION"]
  }
}
```

Maps:
- **Test sets** to capabilities they cover
- **Test cases** to automation entry points
- **Capabilities** to required test sets

Used by TestPlanner to select appropriate tests based on risk.

## Parsing Verdicts Programmatically

```python
import json

with open("artifacts/release_verdict.json") as f:
    verdict = json.load(f)

# Extract decision
decision = verdict["decision"]  # "approve", "block", "needs_review"

# Get risk score
risk_score = verdict["risk"]["score"]  # 0-100
risk_level = verdict["risk"]["level"]  # "low", "medium", "high", "critical"

# Review failures
for failure in verdict["triage"]["failures"]:
    print(f"{failure['test_key']}: {failure['category']} ({failure['confidence']:.1%})")
    print(f"  Reason: {failure['reasoning']}")

# Get next action
print(f"Release verdict: {verdict['decision']}")
print(verdict["release_notes"][0])
```

## CI/CD Integration

Extract verdict in your pipeline:

```bash
#!/bin/bash
set -e

# Run Release Sentinel
python -m releasesentinel run --manifest change_manifest.json --pretty

# Parse verdict
DECISION=$(jq -r '.decision' artifacts/release_verdict.json)

# Gate based on decision
if [ "$DECISION" = "block" ]; then
  echo "❌ Release blocked due to test failures"
  jq '.triage.failures' artifacts/release_verdict.json
  exit 1
elif [ "$DECISION" = "needs_review" ]; then
  echo "⚠️  Release needs human review"
  exit 1
else
  echo "✅ Release approved"
  exit 0
fi
```
